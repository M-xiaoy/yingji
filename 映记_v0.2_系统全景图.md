# 映记 v0.2 系统全景图

> 生成时间：2026-06-30 19:20
> 目的：让你不碰代码就能看懂映记现在能做什么、不能做什么、缺什么

---

## 🗺️ 一句话定位

**映记 = 程序的 AI 化身**

一个程序装上它，就能跟外部 AI 用自然语言对话。外部 AI 不用写 SQL、不用调 REST API、不用读文档——直接说人话就行。

## 📦 所有文件及职能

### 入口层

| 文件 | 职能 | 当前能力边界 |
|------|------|------------|
| `__init__.py` | Yingji 主类，统一入口 | 维护 1 组多轮上下文，单实例 |
| `server.py` | FastAPI HTTP 服务 | 单进程，请求排队，无并发 |
| `use.py` | 小云专用快捷入口 | 通过 HTTP 调 v0.1 API，不经过 AI 对话层 |

### AI 对话层（v0.2 核心新增）

| 文件 | 职能 | 当前能力边界 |
|------|------|------------|
| `_chat.py` | 意图理解+安全检查+回复生成 | LLM 调用同步阻塞，单轮 2-10s |
| `_capability.py` | 能力声明（自我介绍） | 静态 + 动态 Service 扫描 |
| `service.py` | YingjiService 基类 | 定义了接口规范，无 Service 间通信 |

### Service 层

| 文件 | 职能 | 当前能力边界 |
|------|------|------------|
| `services/memory_service.py` | 内置的记忆操作 Service | recall/store/list/delete，安全分级 |
| `service.py` (基类) | Service 的抽象定义 | name/description/operations/execute |

### 存储层

| 文件 | 职能 | 当前能力边界 |
|------|------|------------|
| `store/abc.py` | MemoryBackend 抽象接口 | 6 个方法定义，可插拔 |
| `store/memory_store.py` | SQLite + ChromaDB 双引擎 | WAL 模式，但**写操作没锁，并发会崩** |
| `store/default_backend.py` | 默认后端的封装 | 直接内存调用不走 HTTP |

### 引擎层

| 文件 | 职能 | 当前能力边界 |
|------|------|------------|
| `engine/retriever.py` | 混合检索 v2：向量+BM25+RRF+重排 | 自适应候选池，Cross-encoder 约 2s/次 |
| `engine/reranker.py` | Cross-encoder 重排（bge-reranker-v2-m3） | CPU 可跑，~568MB |
| `engine/extractor.py` | 规则版记忆提取（关键词分类） | 粗略，准确率一般 |
| `engine/llm_extractor.py` | LLM 增强记忆提取 | 调 DeepSeek 提取，精度更高 |
| `engine/compactor.py` | 记忆压缩（去重/降冷） | 基础实现，未测试 |
| `engine/forgetting.py` | 遗忘调度（降冷归档） | 基础实现，无定时器 |
| `engine/phrase_detector.py` | 动态词组检测（自动组词） | 按 n-gram 共现频率自动发现词组 |

### API 层

| 文件 | 职能 | 当前能力边界 |
|------|------|------------|
| `api/routes.py` | v0.1 兼容 REST API（CRUD + 召回） | 单用户，无鉴权 |
| `models.py` | Pydantic 数据模型 | 输入输出格式定义了，**但没做输入校验** |

### 配置

| 文件 | 职能 |
|------|------|
| `config.py` | 所有配置参数（路径、模型名、安全等级、遗忘阈值） |

## 🔍 每条链路的「能」与「不能」

### 链路 1：用户方 AI → 映记 chat()

```
用户方AI: "帮我查一下实习相关的记忆"
映记:     {"reply": "找到了...", "intent": "recall", "data": [...]}
```

| ✅ 能做 | ❌ 不能做 |
|--------|----------|
| 自然语言→结构化操作的意图匹配 | **没有会话隔离** — 多用户同时 chat() 会串上下文 |
| 多轮对话（记得上一轮说了什么） | **不支持流式回复** — 必须等完整 LLM 调用结束 |
| 安全确认（写要确认，删要双重确认） | **没有超时机制** — LLM 卡住就永久阻塞 |
| 能力声明（对方连上来知道我能干嘛） | **没有连接管理** — 对方断开/重连无法感知 |
|                                           | **没有错误恢复** — LLM 返回非 JSON 直接崩 |

### 链路 2：映记 → 内置 MemoryService

```
意图: "recall" → MemoryService.execute("recall", {"query": "实习"})
              → 引擎检索 → 返回结果
```

| ✅ 能做 | ❌ 不能做 |
|--------|----------|
| 三层检索（向量+BM25+重排） | **目前只有 MemoryService 这一个内置** |
| 记忆分层（热/温/冷） | 其他 Service 需要你手动写 |
| 动态组词（自动发现词组） | 检索延迟偏高（重排 2s） |
| 自适应候选池 | |

### 链路 3：映记 → 自定义 Service

```
yj.register(OrderService())
→ 能力声明自动包含 OrderService
→ chat() 能理解"查一下最近的订单"
```

| ✅ 能做 | ❌ 不能做 |
|--------|----------|
| 动态注册新 Service | **没有跨 Service 调用** — Service A 不能调用 Service B |
| 操作 Schema 带类型/必填/枚举/默认值 | **没有 Service 间优先级** — 两个 Service 都能处理同一个 intent 会冲突 |
| 能力声明自动包含新 Service | **没有异步操作** — Service.execute() 必须同步返回 |

## 🧩 多租户场景下的差距分析

你要的拓扑：

```
用户A的AI ──┐
用户B的AI ──┼──→ [同一个映记实例] ──→ [某个程序]
用户C的AI ──┘
```

### 现存代码逐个检查

| 需要的能力 | 现有 | 差距 |
|-----------|------|------|
| **连接管理** — 知道谁连上了、谁断开了 | ❌ 无 | 每个请求独立，无 session 概念 |
| **会话隔离** — A 的上下文跟 B 的不混 | ❌ 无 | `_context` 是 Yingji 实例级的，所有人共享 |
| **并发请求** — A 和 B 同时发消息不会互相等 | ❌ 无 | LLM 调用是同步的，请求排队 |
| **数据隔离** — A 的记忆跟 B 的物理分开 | ❌ 无 | SQLite+ChromaDB 公用，无用户维度 |
| **认证鉴权** — 知道"这个请求是 A 发的" | ❌ 无 | API 没有 token/签名 |
| **限流熔断** — A 刷屏不影响 B | ❌ 无 | 无 |
| **并发安全** — A 删数据时 B 查数据不会崩 | ❌ SQLite 写操作无锁 | `create_memory` 和 `delete_memory` 没有事务隔离 |
| **意图冲突** — 同程序同时收到 A 说"变大" B 说"变小" | ⚠️ 没考虑 | 程序层面需要决定冲突策略 |

### 对，差的有点多

但这不是你代码写得不好——**这是 v0.2 设计时就没考虑多租户**。

v0.2 设计的假设是：**一个程序内嵌一个映记，一个映记只服务一个外部 AI**。

你现在的想法比代码领先了一个版本。

## 🆚 你的想象 vs 代码现实

| 你以为的 | 代码实际 |
|---------|---------|
| `yj.chat("查订单")` 服务所有用户 | 当前是单实例单上下文，多人都用它串上下文 |
| 多个用户同时发给同一个映记 | 请求排队，第二个要等第一个的 LLM 跑完 |
| A 的数据跟 B 的分开 | 所有记忆混在一个 SQLite + 一个 ChromaDB 集合里 |
| 有协议约定 A 的 AI 怎么跟映记对话 | 只有单向 query→reply，没有握手/协商/错误恢复 |

## 🔑 从这里出发：AI-to-AI 协议要写什么

现在你知道缺口在哪了。AI-to-AI 协议至少要包含：

```
1. 连接/握手阶段
   └── 用户方AI: "我是A的AI"
   └── 映记: 返回能力声明 + 建立独立上下文

2. 请求/响应阶段 (现有)
   └── 用户方AI: "帮我查XX"
   └── 映记: 返回 {reply, intent, data}

3. 协商阶段 (新)
   └── 用户方AI: "把XX删掉"
   └── 映记: "XX是哪个？有3条匹配，请指定"
   └── 用户方AI: "ID为xxx的那个"
   └── 映记: "已删除"

4. 会话管理 (新)
   └── 隔离: A 的上下文不影响 B
   └── 超时: 长时间无响应要断开
   └── 重连: 断开的会话能恢复

5. 多租户安全 (新)
   └── 认证: 知道谁是谁
   └── 数据分区: A 只能查 A 的数据
   └── 操作隔离: A 的操作不影响 B
```
